# Project Insight: 基于多智能体协同的深度洞见精炼引擎

**版本:** V0.4 (调研验证版) **状态:** 开发就绪 (Ready for Code Generation) **目标受众:** 核心开发者、AI 编程助手 (如 Claude Code, Cursor)、架构评审委员会

## 1. 项目愿景与核心哲学 (The Philosophy)

本项目旨在构建一个完全由多智能体（Multi-Agent）驱动的个人终端。它不仅是一个内容阅读器，更是一个替用户承担认知负荷的“外脑计算集群”。

- **反头条逻辑与注意力收敛 (Anti-Toutiao & Attention Convergence):** 现代信息流产品的底层逻辑是“用算法榨取用户的停留时间”。本作的核心设计哲学是**“信息终结感”**。每天仅输出固定数量（5-10 张）的极简洞见卡片。拒绝任何形式的无限下拉、推荐流或红点诱惑。用户阅毕即触发“静默状态”，系统将明确告知用户“今日信息已处理完毕，请回归现实生活”。
- **算力换认知 (Compute for Cognition):** 不再让人脑去处理几万字的原始互联网“生肉”。后端将不计成本地利用 16亿 Token 算力进行广度检索、深度洗稿、实体识别与逻辑重构。系统不仅做“摘要”，更做“演绎”——它必须能像资深分析师一样，指出多篇报道背后的根本矛盾或趋势。
- **绝对确定性与零幻觉容忍 (Absolute Determinism):** 传统的 LLM 应用常常死于“结构化输出崩溃”和“Agent 陷入死循环”。本作的系统大模型输出必须是强类型约束的（基于 AST 或 Schema 强校验），多智能体流转必须是基于有向无环图（DAG）或严格状态机（State Machine）可控的。拒绝任何“黑盒式”的概率失败，将大模型的非确定性封锁在极小的内容生成沙盒内。

## 2. 市场竞品分析与差异化定位 (Competitive Landscape)

### 2.1 现有竞品对比

| 产品 | 定位 | Project Insight 差异化优势 |
|------|------|---------------------------|
| **Feedly** | AI RSS 阅读器，强调内容聚合 | Insight 更强调"终结感"，拒绝用户时长成瘾 |
| **Inoreader** | 规则驱动的过滤引擎 | Insight 用 AI 做深度逻辑分析，非简单关键词过滤 |

| **AlphaSense** | 企业级市场情报平台 | Insight 是 C 端个人产品，更轻量、更普惠 |
| **Mem.ai** | AI 笔记聚合工具 | Insight 是"被动摄入"，非主动记录 |

### 2.2 核心竞争壁垒

1. **反头条逻辑 (Anti-Toutiao):** 在"注意力经济"时代，拒绝无限信息流，每日固定 5-10 张卡片，阅毕即停
2. **多智能体对抗质量保证:** CriticNode 红队审稿机制，非简单摘要，而是深度逻辑验证
3. **历史脉络碰撞:** 向量检索历史洞见，发现趋势演变和矛盾，提供时间维度的深度洞察
4. **绝对确定性架构:** LangGraph 状态机 + Instructor 强类型输出，拒绝黑盒式概率失败

## 3. 核心架构与技术栈硬约束 (Strict Tech Stack)

> **⚠️ 对 AI 助手的严格指令：** 必须严格采用以下指定库，禁止降级使用替代方案或自由发挥。以下选型均经过工业级高并发与容错性考量。

### 2.1 后端引擎 (The Backend Engine)

- **API 框架:** `Python 3.11+` / `FastAPI`
  - *Rationale:* FastAPI 提供的原生异步 (`asyncio`) 能力是处理并发爬虫请求和大模型 I/O 延迟的唯一解。其基于 Pydantic 的自动 OpenAPI 文档生成将极大降低与移动端对齐接口的成本。
- **Agent 编排框架:** `LangGraph`
  - *Rationale:* 严禁手写 `while` 循环或使用 LangChain 原生的 AgentExecutor 进行多智能体流转。必须使用 LangGraph 定义严格的 `StateGraph`，设定最大递归深度（Recursion Limit），彻底防止大模型产生幻觉导致的无限工具调用和 Token 燃烧。
  - *调研验证 (2024):* LangGraph 是多智能体编排领域的主流选择，支持 Supervisor、Swarm、Hierarchical Teams 等成熟模式，配套 LangGraph Studio 可视化调试工具。相比 CrewAI（角色扮演，易上手）和 AutoGen（对话驱动，灵活），LangGraph 更适合需要"绝对确定性"的生产环境。
- **LLM 结构化输出:** `Instructor` 配合 `Pydantic V2`
  - *Rationale:* 绝对禁止让模型自行输出 raw JSON 然后手动解析。必须使用 Instructor（通过 `response_model` 注入机制），将 LLM 的自然语言响应在底层强制转换为强类型的 Pydantic 模型，并利用其内置的重试机制 (`max_retries`) 解决格式错误。
  - *调研验证 (2024):* Instructor 是 LLM 结构化输出的事实标准，支持 OpenAI、Anthropic、Google、Mistral 等主流提供商。Field 的 `description` 会注入到 LLM Prompt，显著提升输出质量。推荐使用 `instructor.Mode.TOOLS` 模式（最稳定）。
  - *最佳实践:*
    ```python
    client = instructor.patch(OpenAI(), mode=instructor.Mode.TOOLS)
    result = client.chat.completions.create(
        model="gpt-4o",
        response_model=FinalBriefing,
        max_retries=3,  # 生产环境必须 >= 1
        messages=[...]
    )
    ```
- **任务调度与异步队列:** `Celery` + `Redis`
  - *Rationale:* 每天凌晨的深度萃取任务极为耗时（单次可能长达 15 分钟）。必须通过 Celery 剥离为后台异步任务，Redis 既作为 Broker 也作为分布式锁的后端，防止同一任务被重复触发。
- **持久化关系型数据库:** `PostgreSQL` + `SQLAlchemy 2.0` (async) + `Alembic`
  - *Rationale:* 存储用户配置、API 鉴权和已清洗的结构化卡片。强事务支持是数据一致性的底线。
- **向量数据库与语义检索:** `ChromaDB` (本地化部署)
  - *Rationale:* 使用 `BAAI/bge-m3` 等多语言高精度本地 Embedding 模型。所有精炼后的历史洞见均需进行向量化，为后续的”历史洞察生成”提供跨越时间周期的语义对撞能力。
  - *调研验证 (2024):* ChromaDB 是轻量级开发者友好的向量数据库，适合原型验证和小规模应用（<1M 向量）。性能基准测试显示：
    | 数据库 | QPS (1M 向量) | Recall@10 | 适用场景 |
    |--------|---------------|-----------|----------|
    | **ChromaDB** | 500-2,000 | 95%+ | 原型/小规模 |
    | **Qdrant** | 5,000-15,000 | 98%+ | 生产环境，高性能 |
    | **Milvus** | 5,000-20,000 | 97%+ | 大规模分布式 |
  - *扩展建议:* 
    - **Phase 1-2:** 继续使用 ChromaDB，快速验证概念
    - **生产化阶段:** 设计抽象层（Repository Pattern），便于未来迁移到 Qdrant 或 Milvus
    - **触发迁移条件:** 当历史洞见向量超过 50 万条，或查询延迟 > 200ms 时

### 2.2 移动端前台 (The Android Client)

- **UI 框架:** Kotlin / `Jetpack Compose`
  - *Rationale:* 抛弃沉重的 XML。纯代码声明式 UI 是实现极简设计、复杂物理阻尼动画和动态主题切换的最佳方案。
- **架构范式:** 严格的 `MVI (Model-View-Intent)` 与单向数据流 (UDF)
  - *Rationale:* View 只能发送 Intent 并被动观察 State。彻底阻断双向绑定的脏状态问题，保证界面的可预测性。
  - *调研验证 (2024):* MVI + Compose 是 Android 现代架构的主流范式，符合 Google 官方推荐。关键最佳实践：
    - 使用 `collectAsStateWithLifecycle()` 替代 `collectAsState()`，确保生命周期感知
    - Sealed Interface 定义状态/意图，保证类型安全
    - Side Effect 通过 `Channel` 或 `SharedFlow` 处理一次性事件
- **并发与响应式状态:** `Kotlin Coroutines` + `StateFlow`
  - *Rationale:* `StateFlow` 提供了防抖（State conflation）和生命周期感知的收集机制 (`collectAsStateWithLifecycle`)，避免内存泄漏。
- **依赖注入 (DI):** `Dagger Hilt`
  - *Rationale:* 强制解耦网络层、数据库层与表现层，为未来的多端扩展或组件替换提供工程冗余。
- **网络与本地持久化 (Offline-First):** `Retrofit2` / `Room Database`
  - *Rationale:* 遵循离线优先原则。每天清晨拉取数据后存入 Room 数据库，用户在地铁断网等极端弱网环境下依然能获得毫秒级的丝滑阅读体验。

## 3. 多智能体工作流 (LangGraph State Machine)

必须使用 LangGraph 定义系统状态。状态不仅是数据的载体，更是多个智能体之间沟通的“共享白板”（Blackboard Pattern）。

### 3.1 状态数据结构定义 (State Schema)

```
from typing import TypedDict, Annotated, List
import operator

class GraphState(TypedDict):
    target_topic: str
    raw_texts: Annotated[List[str], operator.add] # 允许累加抓取的文本
    draft_insights: List[dict]
    critic_feedback: List[str]
    final_briefings: List[dict]
    retry_count: int
```

### 3.2 节点定义与流转逻辑

- **节点 1：ScoutNode (星海寻路人 - 广度检索)**
  - **触发:** 接收调度系统的 Topic 列表。
  - **行为:** 根据 Topic 动态推演搜索关键词。调用 `search_web` 和 `fetch_webpage_content`。必须具备简单的绕过反爬机制（如设置 User-Agent、处理简单 JS 渲染）。
  - **产出:** 将提取的纯文本片段追加至 `raw_texts`。
- **节点 2：DistillerNode (提纯者 - 深度清洗)**
  - **行为:** 阅读数万字的 `raw_texts`。剥离营销修辞、情绪化词汇。其 System Prompt 必须设定为：“你是一个冷酷客观的分析师，只提取事实、数据和逻辑链条”。
  - **产出:** 通过 Instructor 强制输出 `DraftInsight` 对象数组。
- **节点 3：CriticNode (红队审稿人 - 交叉验证)**
  - **行为:** 作为系统内最核心的“质量门神”。逐条审视 `DraftInsight`。检查逻辑跳跃、数据孤证或明显的事实冲突。
  - **路由机制 (Conditional Edge):**
    - *失败 (Rejected):* 如果发现某条洞见结论缺乏支撑，将尖锐的修改意见写入 `critic_feedback`，`retry_count += 1`，将状态路由退回 `ScoutNode` (要求搜集更多证据) 或 `DistillerNode` (要求重写)。设定硬限制 `retry_count <= 3`，超时则抛弃该条线索。
    - *成功 (Approved):* 清空反馈，状态进入下一节点。
- **节点 4：MemoryEditorNode (主编与记忆映射 - 长期价值构建)**
  - **行为:** 对 Approved 的洞见执行 Embedding。向 ChromaDB 查询相似度 (Cosine Similarity) 超过 0.82 的历史节点。
  - **整合:** 如果发现某条关于“固态电池”的新闻与半年前的另一条新闻形成互斥或递进关系，主编 Agent 必须生成一段 `historical_insight`。
  - **落库:** 生成最终的 `FinalBriefing`，写入 PostgreSQL，同步更新新的 Embedding 到 ChromaDB。

## 4. LLM 数据契约 (Pydantic Models)

> **⚠️ 对 AI 助手的严格指令：** 必须使用以下 Pydantic 模型约束大模型的输出。Field 中的 `description` 是 Instructor 引导大模型输出的核心 Prompt，必须保留且严格遵循。

```
from pydantic import BaseModel, Field, field_validator
from typing import List, Optional
import datetime

class Source(BaseModel):
    name: str = Field(..., description="信息来源名称，如 'Arxiv', 'TechCrunch', 'GitHub Repo'")
    url: str = Field(..., description="原文绝对链接，必须是合法的 URL 格式")
    
    @field_validator('url')
    def validate_url(cls, v):
        if not v.startswith('http'):
            raise ValueError("URL 必须包含 http/https 协议头")
        return v

class LogicPoint(BaseModel):
    premise: str = Field(..., description="推演前提或客观数据支撑")
    conclusion: str = Field(..., description="基于该前提得出的中间结论")

class FinalBriefing(BaseModel):
    date_generated: datetime.date = Field(default_factory=datetime.date.today)
    category: str = Field(..., description="高维分类标签，最多三个单词，如 'AI Engineering' 或 'Quantum Computing'")
    core_thesis: str = Field(
        ..., 
        description="一句话概括核心论点，极其精炼，禁止使用形容词堆砌，字数控制在 40 字以内"
    )
    logic_chain: List[LogicPoint] = Field(
        ..., 
        description="支撑核心论点的严密逻辑推演链，3-5个节点，层层递进",
        min_length=2, 
        max_length=5
    )
    historical_insight: Optional[str] = Field(
        default=None, 
        description="系统检测到的历史知识脉络碰撞。如果此事件验证或推翻了历史记录，请指出来。若无关联则严格返回 null"
    )
    sources: List[Source]
```

## 5. Android MVI 架构与状态机契约

> **⚠️ 对 AI 助手的严格指令：** ViewModel 必须且只能暴露 `StateFlow<ViewState>`，通过 channel 暴露一次性 `Effect`。严格遵守如下状态定义，禁止在 ViewModel 之外直接修改状态。

```
// 1. 状态 (UI 渲染的唯一真相来源，代表当前界面的定格画面)
sealed interface InsightState {
    data object Loading : InsightState // 初始化加载，渲染全屏极简骨架屏
    data class Success(
        val cards: List<BriefingCard>,
        val currentIndex: Int = 0,
        val isOfflineMode: Boolean = false // 网络断开时，读取 Room 数据库数据

    data object AllCaughtUp : InsightState // 终结态：阅毕状态，展示科技留白页面
}

// 2. 意图 (用户在界面的显式操作，驱动状态变更)
sealed interface InsightIntent {
    data object LoadTodayBriefings : InsightIntent
    data class CardSwipedAway(val cardId: String) : InsightIntent
    data class ToggleSourceDetails(val cardId: String) : InsightIntent
    data object RequestRetry : InsightIntent
}

// 3. 一次性副作用 (Side Effects，用于一次性消费，如触觉反馈、Toast、页面跳转)
sealed interface InsightEffect {
    data class ShowToast(val message: String) : InsightEffect
    data object TriggerHapticFeedback : InsightEffect // 触发物理震动马达
    data class NavigateToBrowser(val url: String) : InsightEffect

```

## 6. 移动端 UI/UX 与物理交互规范 (Compose 指南)

UI 绝不只是排版，它是系统哲学的外显。本作要求达到“顶级数码硬件官网”或“独立高级画廊”般的视觉质感。

- **极致米白色彩系统 (Off-White/Cream Mode First):**
  - 主背景强制锁定为极具质感的米白色或纸张色 (如 `#F5F5F0` 或 `#F9F9F6`)，营造出独立高级画廊或顶级实体刊物的呼吸感。
  - 卡片背景使用纯净的本白 (`#FFFFFF`)，并通过极其微弱的弥散阴影 (`Elevation`) 与背景拉开空间层次。
  - 摒弃传统的彩色分类标签。大面积正文使用极其深邃的炭黑色 (如 `#1A1A1A` 或 `#2C2C2C`) 保证沉浸式阅读，次级文字使用高级中性灰 (如 `#8C8C8C`)，形成干净利落的对比。
  - **点睛之笔 (Accent Color):** 全局仅保留一种强烈的强调色（例如极其克制的克莱因蓝 `#002FA7` 或爱马仕橙 `#F37021` ），仅用于极其稀少的交互焦点（如外链图标或特殊的“历史洞察”指示灯）。
- **空间留白与排版 (Spatial Typography):**
  - 摒弃紧凑感。卡片内边距 (Padding) 统一加倍（推荐基准为 `24dp` 或 `32dp`）。
  - 标题使用极其粗壮的无衬线字体 (Sans-Serif, Black/ExtraBold)，正文使用纤细的等宽或标准字体 (Regular/Light)，形成强烈的字重反差。
- **物理感阻尼交互 (Physical Damping Interaction):**
  - 采用 `HorizontalPager` 实现卡片的沉浸式单张翻页。
  - 卡片滑出屏幕时，必须计算滑动速率 (Velocity)。配合 `spring` 物理动画曲线，滑动时给予顺滑的非线性阻尼反馈。
  - **触觉反馈引擎 (Haptics):** 在卡片翻页锁定瞬间、阅毕状态触发瞬间，强制调用设备的震动马达（HapticFeedbackType.LongPress 或定制 Waveform），提供极其克制但有力的物理确认感。
- **极简终结态 (The "All Caught Up" State):**
  - 用户划走最后一张卡片时，彻底隐藏所有多余的 UI 框架。
  - 屏幕正中央缓慢淡入 (Fade-in, duration=1000ms) 极简的一行字：“今日洞见已凝结完毕”。
  - 无加载更多按钮，无返回按钮，强制用户结束摄入，回归生活。

## 7. 风险评估与成本控制 (Risk Assessment & Cost Control)

> **⚠️ 对 AI 助手的严格指令：** 以下风险点必须在开发过程中持续关注，并在对应 Phase 中实现缓解措施。

### 7.1 高风险项

#### 🔴 Token 成本失控风险
- **问题:** 文档提到"不计成本利用 16 亿 Token"，这在生产环境中不可持续。GPT-4o 的 Input Token 成本约 $2.5/1M，Output Token 约 $10/1M。
- **缓解措施:**
  1. 增加成本监控层，记录每次 LLM 调用的 Token 消耗
  2. 设置每日 Token 消耗上限（建议初期 $50/天）
  3. 优化 Prompt 长度，避免重复传递历史上下文
  4. 使用 Prompt Caching（如 Anthropic API）减少重复计算
- **实施阶段:** Phase 1

#### 🔴 反爬虫法律合规风险
- **问题:** ScoutNode 需要"绕过反爬机制"，这可能违反网站服务条款或相关法律。
- **缓解措施:**
  1. 优先使用官方 API（如 News API、Google News API、GitHub API）
  2. 遵守 robots.txt 规则
  3. 设置合理的请求频率限制（建议 < 1 req/s/domain）
  4. 记录所有爬取行为，便于审计
- **实施阶段:** Phase 1

#### 🔴 LLM 幻觉与事实核查风险
- **问题:** 即使有 CriticNode，LLM 仍可能生成看似合理但错误的洞见。
- **缓解措施:**
  1. 在 `FinalBriefing` 模型中增加 `confidence_score: float` 字段（0-1）
  2. 低置信度（< 0.7）内容需标记为"待验证"
  3. 要求所有数据点必须有至少两个独立来源支撑
  4. CriticNode 的 System Prompt 必须强调"无证据则返回 null"
- **实施阶段:** Phase 2

### 7.2 中风险项

#### 🟡 ChromaDB 扩展性瓶颈
- **问题:** ChromaDB 单节点架构，百万级向量时性能可能成为瓶颈。
- **缓解措施:**
  1. 设计抽象层（Repository Pattern），隔离向量数据库实现细节
  2. 定期监控查询延迟，设置告警阈值（> 200ms）
  3. 准备迁移方案：ChromaDB → Qdrant（推荐）或 Milvus
- **实施阶段:** Phase 2

#### 🟡 Android 离线同步冲突
- **问题:** Room + Retrofit 的离线优先设计需要处理数据冲突。
- **缓解措施:**
  1. 初期采用"最后写入胜出"（Last Write Wins）策略
  2. 每条记录增加 `updated_at` 时间戳
  3. 仅同步增量数据，减少冲突概率
- **实施阶段:** Phase 3

### 7.3 低风险项

#### 🟢 UI 动画性能
- **问题:** 60fps 物理阻尼动画在低端设备上可能卡顿。
- **缓解措施:**
  1. 使用 `Modifier.graphicsLayer` 硬件加速
  2. 避免在动画过程中进行重组（Recomposition）
  3. 真机测试覆盖低端设备（如 2GB RAM）
- **实施阶段:** Phase 4

## 8. 执行步骤与 CI/CD 拆解 (Roadmap for AI Assistant)

请开发 AI 助手按照以下阶段严格步进。每一阶段的核心逻辑必须配备单元测试 (Unit Tests)。在未通过所有断言之前，禁止进入下一阶段。

- **Phase 1: 核心 LLM 与数据通道测试 (Backend Core)**
  - **优先级调整 (基于调研):** Instructor + Pydantic 验证是整个项目的技术基石，必须首先完成。
  - **Step 1.1:** 配置 FastAPI 骨架。
  - **Step 1.2:** 利用 `pytest` 编写测试，模拟输入一篇充满废话的新闻稿，验证 `Instructor` 能否绝对稳定地将其转化为符合上述 `FinalBriefing` Schema 的强类型 JSON。
    - **验收标准:** 连续 100 次调用，成功率 >= 99%
    - **测试用例:** 包含极端情况（空文本、超长文本、多语言混合、包含特殊字符）
  - **Step 1.3:** 实现 Token 成本监控层，记录每次 LLM 调用的 Token 消耗。
  - **Step 1.4:** 搭建本地 PostgreSQL + Alembic 迁移脚本，确保数据表结构创建成功。
  - **Step 1.5:** 实现基础的 ScoutNode，使用官方 API（News API）而非直接爬取，确保法律合规。
- **Phase 2: LangGraph 多智能体对抗引擎 (Multi-Agent Engine)**
  - **Step 2.1:** 定义 Graph State。
  - **Step 2.2:** 组装 Scout -> Distiller -> Critic -> Editor 的有向无环图。
  - **Step 2.3:** 为 FinalBriefing 模型增加 `confidence_score: float` 字段，CriticNode 必须输出置信度评分。
  - **Step 2.4:** **极限测试:** 故意注入充满逻辑矛盾的假数据，验证 CriticNode 能否成功触发阻断逻辑，并将重试计次 (`retry_count`) 正确递增。
  - **Step 2.5:** 设计向量数据库抽象层（Repository Pattern），隔离 ChromaDB 实现细节，便于未来迁移。
  - **Step 2.6:** 联调 ChromaDB，验证向量检索功能。监控查询延迟，设置告警阈值（> 200ms）。
- **Phase 3: Android 核心业务与状态机驱动 (Android Architecture)**
  - **Step 3.1:** 搭建 Hilt DI 图。
  - **Step 3.2:** 建立 Room 数据库层，实现与 Retrofit 接口层的数据同步逻辑 (Repository Pattern)。
  - **Step 3.3:** 实现离线同步策略：采用”最后写入胜出”（Last Write Wins）策略，每条记录增加 `updated_at` 时间戳。
  - **Step 3.4:** 利用 `JUnit` + [Turbine](https://github.com/cashapp/turbine) 测试 ViewModel，输入一系列假 `Intent`，验证 `StateFlow` 是否按照状态机规则输出预期的 `InsightState`。
  - **验收标准:** 状态转换覆盖率 100%，包括边界情况（网络断开、空数据、加载失败）。
- **Phase 4: Compose 极简美学渲染 (UI/UX Realization)**
  - **Step 4.1:** 严格遵守”米白底色、强反差字重、极致留白”规范完成 UI 开发。
  - **Step 4.2:** 重点死磕手势处理与物理阻尼动画。使用 `Modifier.graphicsLayer` 硬件加速，避免动画过程中重组。
  - **Step 4.3:** 实现触觉反馈引擎（Haptics），在卡片翻页锁定瞬间调用 `HapticFeedbackType.LongPress`。
  - **验收标准:** 在模拟器或真机上确保横向滑动卡片时的流畅度达到 60fps，且无任何掉帧与 UI 撕裂。低端设备（2GB RAM）测试通过。
